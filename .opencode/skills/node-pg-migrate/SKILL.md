# node-pg-migrate: Safe Migration Generator

## Goal

Generate safe, reversible PostgreSQL migrations using node-pg-migrate.

---

## Context

* Database: PostgreSQL
* Migration tool: node-pg-migrate
* Naming convention: snake_case
* IDs: UUID type, we will use uuidv7 , this will be generated in the application layer, not in the database.
* Enums: implemented as TEXT + CHECK (not Postgres ENUM), SCREAMING_SNAKE_CASE 

---

## Core Rules

### 0. Always confirm before running migrations

**Never run `migrate:up`, `migrate:down` without first asking the user for confirmation.** This applies even if the user explicitly requests a migration to be run — pause and confirm first.

### 1. Always reversible

* Every migration MUST include both `up` and `down`
* `down` must fully undo `up`

---

### 2. Never destructive without fallback

Avoid:

* dropping columns
* renaming columns
* changing types directly

Instead:

* create new column
* backfill data
* switch usage
* optionally drop later (separate migration)

---

### 3. Use TEXT + CHECK instead of ENUM

```ts
pgm.addConstraint("users", "users_status_check", {
  check: "status IN ('DRAFT', 'PUBLISHED')",
})
```

---

### 4. Always use snake_case

```ts
tableName: users
columnName: created_at
```

---

### 5. Timestamps

```ts
created_at: { type: "timestamp", notNull: true, default: pgm.func("now()") }
updated_at: { type: "timestamp", notNull: true, default: pgm.func("now()") }
```

---

### 6. IDs

```ts
id: { type: "uuid", primaryKey: true }
```

### 7. Never edit the generated schema file

`schema.generated.ts` is auto-generated by `kysely-codegen`. **Never edit it manually.**

If you need to override a type (e.g. replace `string` with an enum type), use `schema.override.ts` instead.

**Correct workflow after adding a migration:**

1. Create the migration file using `pnpm migrate:create <name>` (generates `.ts` files)
2. Edit the generated file to add `up` and `down`
3. Run `pnpm migrate:up` against the database
4. Run `pnpm gen` to regenerate `schema.generated.ts`
5. If needed, update `schema.override.ts` to patch specific types

**Note:** Migration files are `.ts`. The project's `migrate:up`, `migrate:down`, and `migrate:create` scripts all include `-j ts` — node-pg-migrate loads `.ts` files natively. No tsx/ts-node wrapper needed.

---

### Ask user: amend last migration or create new one

When a schema change is needed, ask the user which approach they prefer:

**Option A — Amend the last migration** (avoids creating many files):
1. Run `pnpm migrate:down` to rollback the last migration
2. Update the last migration file with the revised `up` / `down`
3. Run `pnpm migrate:up` to apply the updated migration
4. Run `pnpm gen` to regenerate `schema.generated.ts`

**Option B — Create a new migration file** (see below).

---

### Creating a new migration

Always use the `migrate:create` command — it generates a file with a timestamp prefix automatically, with `-j ts` producing `.ts` output:

```bash
pnpm migrate:create <snake_case_description>
```

Example:
```bash
pnpm migrate:create add_user_table
# → migrations/20260523120000_add_user_table.ts
```

The file name is generated by node-pg-migrate using its default naming convention. Do not create migration files manually or rename them after creation.

## Patterns

---

### 🧱 Create Table

```ts
export async function up(pgm) {
  pgm.createTable("users", {
    id: { type: "uuid", primaryKey: true },
    email: { type: "text", notNull: true },
    status: { type: "text", notNull: true, default: "DRAFT" },
    created_at: {
      type: "timestamp",
      notNull: true,
      default: pgm.func("now()"),
    },
    updated_at: {
      type: "timestamp",
      notNull: true,
      default: pgm.func("now()"),
    },
  })

  pgm.addConstraint("users", "users_status_check", {
    check: "status IN ('DRAFT', 'PUBLISHED')",
  })
}

export async function down(pgm) {
  pgm.dropTable("users")
}
```

---

### ➕ Add Column (safe)

```ts
export async function up(pgm) {
  pgm.addColumn("users", {
    phone_number: { type: "text" },
  })
}

export async function down(pgm) {
  pgm.dropColumn("users", "phone_number")
}
```

---

### ⚠️ Add NOT NULL column (safe pattern)

```ts
export async function up(pgm) {
  // 1. Add nullable column
  pgm.addColumn("users", {
    phone_number: { type: "text" },
  })

  // 2. Backfill
  pgm.sql(`
    UPDATE users
    SET phone_number = ''
    WHERE phone_number IS NULL
  `)

  // 3. Add constraint
  pgm.alterColumn("users", "phone_number", {
    notNull: true,
  })
}

export async function down(pgm) {
  pgm.dropColumn("users", "phone_number")
}
```

---

### 🔄 Change enum values (CHECK constraint)

```ts
export async function up(pgm) {
  pgm.dropConstraint("users", "users_status_check")

  pgm.addConstraint("users", "users_status_check", {
    check: "status IN ('DRAFT', 'PUBLISHED', 'ARCHIVED')",
  })
}

export async function down(pgm) {
  pgm.dropConstraint("users", "users_status_check")

  pgm.addConstraint("users", "users_status_check", {
    check: "status IN ('DRAFT', 'PUBLISHED')",
  })
}
```

---

## Anti-Patterns (DO NOT DO)

❌ Direct type changes
❌ Dropping columns in same migration as replacement
❌ Using Postgres ENUM type
❌ Adding NOT NULL column without backfill
❌ Using camelCase in DB

---

## Output Requirements

* Always include `up` and `down`
* Use clear constraint names: `{table}_{column}_check`
* Use raw SQL only when necessary

---

## Optional Enhancements

* Add indexes for frequently queried columns
* Add unique constraints when required

```ts
pgm.createIndex("users", "email", { unique: true })
```

---

## Mental Model

Migrations must be:

* safe to run in production
* reversible
* backward-compatible during rollout
